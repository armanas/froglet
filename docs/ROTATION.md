# Key and Identity Rotation Runbook

Status: **living document**. The doc half of [TODO.md Order 62](../TODO.md)
is closeable via this runbook. The "rehearsal on a non-production instance"
half is explicitly marked as **PENDING HUMAN ACTION** below and keeps Order

Rotation is the backstop for credential compromise, contributor turnover,
or operator-hygiene cadence. A compromise response where the operator has
no documented rotation procedure is a multi-hour outage at best. This
runbook lists every secret the hosted environment holds and the steps to
rotate each without taking the public path down (or at worst a brief
window, flagged explicitly).

## 2. When to rotate

- **Compromise suspected** — any credible signal that a secret may have leaked. Rotate immediately; do not wait for "confirmation."
- **Contributor turnover** — when someone with access to the Keychain backup, 1Password vault, or deployment host leaves or loses authorization.
- **Routine cadence** — every 90 days for Cloudflare token, AWS key, and Voltage macaroon. Calendar reminder.
- **After incident response** — even if the rotated credential wasn't the attack vector. Reduces the uncertainty window.

## 3. Per-secret rotation

### 3.1 Cloudflare DNS token

1. Cloudflare dashboard → https://dash.cloudflare.com/profile/api-tokens
2. Find the existing `Edit zone DNS — froglet.dev` token → **Roll** (or **Delete** + create new).
3. Copy the new token value (shown once).
4. Update Keychain:
   ```bash
   security add-generic-password -a froglet -s cloudflare-dns-token -w '<new-token>' -U
   ```
5. Verify:
   ```bash
   ./scripts/cloudflare_dns.sh verify
   ```
6. If the old token was issued with a TTL, it auto-expires. Otherwise the old token remains valid until you delete it — do the delete step to close the window.

**Downtime:** none. No running service uses this token.

### 3.2 AWS IAM access key

1. AWS Console → IAM → Users → `froglet-deploy` → Security credentials.
2. **Create access key** (second key slot; IAM allows two active keys per user).
3. Update Keychain with the new key pair:
   ```bash
    read -rs AK && security add-generic-password -a froglet -s aws-deploy-access-key -w "$AK" -U && unset AK
    read -rs SK && security add-generic-password -a froglet -s aws-deploy-secret-key -w "$SK" -U && unset SK
   ```
4. Verify:
   ```bash
   ./scripts/deploy_aws.sh verify
   ```
5. Run one benign read-only API call against Lightsail:
   ```bash
   ./scripts/deploy_aws.sh status
   ```
6. Return to IAM → Users → `froglet-deploy` → Security credentials → **Deactivate** the old access key.
7. After 24 hours of the old key being deactivated without any issues, **Delete** it.

**Downtime:** none. Keychain update between step 2 and 6 means any running
helper invocation picks up the new key on its next run; there is no
persistent connection to re-authenticate.

### 3.3 Voltage macaroon

2. Update Keychain:

    read -rs M && security add-generic-password -a froglet -s voltage-admin-macaroon -w "$M" -U && unset M

3. The Froglet node reads the macaroon via env var at deploy time. Re-deploy:
   ```bash
   ./scripts/deploy_aws.sh deploy ghcr.io/armanas/froglet-provider:<current-tag>
   ```
4. Verify Lightning status via the live node:
   ```bash

   curl -sS https://ai.froglet.dev/v1/runtime/wallet/balance -H "Authorization: Bearer $(cat /path/to/runtime.token)"
   ```

**Downtime:** ~2 min during the `deploy` swap; Lightsail keeps the old
deployment serving until the new one is healthy.

### 3.5 Froglet node identity key

**This is the sensitive one.** The node identity key signs every artifact
the node produces (offers, descriptors, receipts). Rotating it means every
downstream consumer that cached the old pubkey thinks this node is a new
provider. For the hosted reference instance at `ai.froglet.dev` that's
currently tolerable because there are no long-standing deals indexed
against it, but for a mature provider this would be a serious product
consequence.

**Procedure (when rotation is unavoidable):**

1. Stop the Lightsail deployment (deploy an empty-config variant or scale
   to 0 — note that Lightsail Container Service scale minimum is 1, so
   you instead `destroy` + re-provision; see §3.5.1 for the alternative).
2. Generate a new identity seed locally:
   ```bash
   openssl rand -hex 32 > /tmp/new-identity.seed
   ```
3. Replace the seed in the on-disk `runtime/identity/` location (the
   Lightsail container mounts a tempdir, so this means replacing the env
   var or re-provisioning with a fresh seed).
4. Deploy the new container. The node generates a fresh `node_id` from the
   seed on startup.
5. Publish a **new identity artifact** so consumers can re-discover.
6. Broadcast the rotation to any downstream marketplace integrations.

#### 3.5.1 Lightsail-specific caveat

Lightsail Container Service doesn't give us a mutable persistent volume —
container filesystem is ephemeral per deployment. That means identity-seed
rotation is driven by **replacing the seed env var**, not by editing a
file inside a running container. Wire the seed into
`ops/lightsail/froglet-node.template.json` as an env var (not currently
done — TODO to wire the existing node's seed-from-env support into the
template).

**Downtime:** ~5 min during the re-provision window.

### 3.6 Runtime + provider-control auth tokens

These are on-disk token files the node generates at first start. Rotation
is done by deleting them from the container filesystem and restarting, at
which point the node regenerates them. Because they live on the ephemeral
Lightsail container filesystem, **each `deploy` already effectively
rotates them** — there is no long-lived token across deployments unless
the operator explicitly persists them.

**Downtime:** <60 s during restart.

### 3.7 GitHub tokens

Not used by the current deploy automation. The release workflow uses
`GITHUB_TOKEN` provided by Actions, which is scoped to the workflow run
and auto-revoked on completion. If a PAT is introduced later for e.g.
marketplace-service repo access, rotate per GitHub's standard rotation:
Settings → Developer settings → Personal access tokens → regenerate.

## 4. Rehearsal — PENDING HUMAN ACTION

Order 62's DoD explicitly requires the rehearsal to be performed on a
**non-production** instance. This runbook doesn't automate that; it cannot
autonomously spin up a throwaway AWS account + Cloudflare zone + Voltage
node to rehearse against.

**What a rehearsal looks like:**

1. Provision a second `froglet-node-rehearsal` Lightsail Container Service
   in the same AWS account (or a throwaway account). `./scripts/deploy_aws.sh create` with a different `FROGLET_AWS_SERVICE_NAME`.
2. Rotate each of §3.1, §3.2, §3.6 against the rehearsal instance, timing
   each.
3. Exercise §3.5 (identity rotation) on the rehearsal instance. Confirm
   the new identity is observable via `/v1/node/identity`.
4. Tear down the rehearsal instance.
5. Record the timings in a rehearsal log appended to this doc.

Until the rehearsal is complete and its log is appended, Order 62 stays
🟡.

## 5. What's deliberately out of scope

- **HSM / KMS integration.** Moving the identity key to a hardware security
  module is a post-MVP change; this runbook assumes on-disk seeds managed
  by the operator.
- **Automated rotation.** The Cloudflare token and AWS keys could be
  rotated on a schedule via a cron-wrapped script, but the tradeoff (extra
  machinery, extra failure modes, extra complexity) is not justified at a
  single-operator alpha. Routine-cadence rotation is calendar-driven.
- **Multi-operator credential handoff.** When there is more than one
  operator, a password manager with shared vaults (1Password Business,
  etc.) becomes the right tool. Single operator today — Keychain is
  sufficient.
